# 明瞭性チェックリスト

スライド作成後、自己採点の前に実施する。

## 1. 用語の明確化

- 専門用語は初出で 1 行定義を付ける
- 略語は原則使わず正式名称で書く。広く知られた略語（CI, API, URL, DB, OS 等）のみ略語可
- スライド内で略語を使う場合でも、初出では必ず正式名称を併記する
- **テーブルセルの略語・技術名は、列ヘッダーのカテゴリに対して自明な値のみ使う。** カテゴリと値の対応が不明瞭な場合は、補足を括弧で付けるか正式名称に置き換える（例: OS 列に「LXD」→「Linux (LXD)」、ランナー列に「cycloud」→ セルフホスト名そのままで OK だが OS 列には実際の OS を書く）
- 1 スライドのフットノートは最大 3 件に抑える

Before/After:

| Before (NG) | After (OK) | 理由 |
|---|---|---|
| `UTF と統合されている` | `Unity Test Framework と統合されている` | 知名度の低い略語。正式名称で書く |
| `SRS のレベルに応じて間隔を変える` | `SRS（間隔反復システム）のレベルに応じて復習間隔を変える` | 略語の初出で正式名称がない |
| `ORM を使ってクエリを書く` | `ORM<sup>※1</sup> を使ってクエリを書く` + フットノートで定義 | 聴衆が知らない可能性のある用語 |
| OS 列に `LXD` | OS 列に `Linux (LXD)` | LXD は OS ではなくコンテナ技術。列ヘッダーが求めるカテゴリに合う値にする |

## 2. 文の読みやすさ

- 1 文はできるだけ 60 文字以内にする
- 1 箇条書きは 1 主張に絞る
- 接続詞を多用しない（「しかし」「また」「つまり」が連続しない）

Before/After:

| Before (NG) | After (OK) | 理由 |
|---|---|---|
| `このライブラリはバンドルサイズが小さく、また型安全で、さらにTree Shakingにも対応している` | `バンドルサイズ: 12KB（gzip）` / `型安全: generics 対応` / `Tree Shaking 対応` | 1 文に 3 主張。箇条書きに分離 |
| `しかし、パフォーマンスの問題がある。しかし、回避策もある。` | `パフォーマンスの課題: 初回描画が 200ms 遅い。回避策: 遅延読み込みで 50ms に短縮` | 「しかし」連続。具体的な数値で書き直す |

## 3. 抽象語の具体化

以下の抽象語を見つけたら、具体語に置換する:

- 最適化
- 効率化
- 柔軟性
- 拡張性
- 高品質

置換パターン:

- 抽象語 + なる -> 何がどう変わるかを数値・条件で書く
- 便利 -> どの作業時間が何分減るかを書く
- 改善 -> どの指標がどれだけ改善するかを書く

Before/After:

| Before (NG) | After (OK) | 理由 |
|---|---|---|
| `パフォーマンスを最適化する` | `API 応答時間を 800ms → 200ms に短縮する` | 「最適化」が何を指すか不明 |
| `開発効率が向上する` | `ホットリロードで保存→反映が 3 秒→0.5 秒になる` | 「効率」を体感できる数値に |
| `柔軟な設計になっている` | `設定ファイルを差し替えるだけで DB を PostgreSQL から MySQL に切り替えられる` | 「柔軟」の中身を操作レベルで説明 |
| `高品質なコードを保つ` | `lint + 型チェック + テストカバレッジ 80% を CI で自動検証する` | 「高品質」の判定基準を明示 |
| `拡張性のあるアーキテクチャ` | `新しい認証プロバイダーは adapter を 1 ファイル追加するだけで対応できる` | 「拡張性」を具体的な追加手順で説明 |

## 4. 見出しの説明力

- 見出しだけ読んで流れが追えるか確認する
- 「概要」「詳細」「補足」のような汎用見出しを避ける
- 見出しは可能なら結論形にする（例: `X を採用すると Y のコストが下がる`）

Before/After:

| Before (NG) | After (OK) | 理由 |
|---|---|---|
| `テスト戦略` | `単体テスト + E2E で変更時の回帰を 5 分以内に検出する` | 汎用見出し。何を達成するか不明 |
| `キャッシュについて` | `Redis キャッシュで DB 問い合わせを 90% 削減する` | 「〜について」は主張がない |
| `まとめ` | `3 つの設計判断がビルド時間を半減させた` | まとめスライドでも結論を見出しに |
| `認証の仕組み` | `JWT + リフレッシュトークンでセッションを 7 日間維持する` | 仕組みの具体的な効果まで含める |

## 5. So What の明示

各スライドの末尾に `▶` プレフィックスで「なぜ重要か」を 1 行で置く。

- 何がわかったか / なぜ重要か を 1 文に凝縮する
- テキストプレフィックス（「つまり」「要するに」等）は使わない — 多用すると単調になる
- `▶` は視覚的な区切りとして機能し、コンテンツ内の矢印 `→` と混同しない
- **▶ 行に本文で触れていない概念を持ち込まない** — ▶ 行はスライド本文の「まとめ」であり、新情報の導入箇所ではない。▶ 行に登場するキーワードがすべて本文（見出し・テーブル・箇条書き・コード）に含まれているか確認する

Before/After:

| Before (NG) | After (OK) | 理由 |
|---|---|---|
| （So What 行なし。コードだけ表示して終わり） | `▶ useEffect のクリーンアップでメモリリークを防げる。` | コードの意味が不明 |
| `つまり便利です。` | `▶ スキーマ共有で API 不整合が本番前に検出される。` | 「便利」は具体性ゼロ |
| `つまり: 依存を一方向にする。` が 10 ページ連続 | 各ページで具体的な効果を `▶` で書く | 同じプレフィックスでも内容が具体的なら単調にならない |
| 表に iOS/Android しかないのに `▶ Web エディタも同一リポジトリで管理` | 表に Web エディタを追加してから ▶ 行で言及 | ▶ 行に本文で未登場の概念がある |

## 6. 矢印表記の整合性

スライド本文中の `→` や Mermaid 図の `-->` は、意味が一貫しているか確認する。

### 依存関係を示す矢印

`A → B` は「A が B に依存する」を意味する。Mermaid 図とテキストの両方でこの規約に従う。

| チェック項目 | NG 例 | OK 例 |
|---|---|---|
| Mermaid 図の矢印方向 | `core --> api_client`（core が api_client に依存？） | `api_client --> core`（api_client が core に依存） |
| テキスト中の `→` | `core → apps の単方向設計` | `apps → core の単方向設計` |
| 図とテキストの不一致 | 図: `mobile --> core` / 本文: `core → mobile` | 図もテキストも同じ方向 |

### 検証手順

1. Mermaid 図がある場合、各 `-->` が「依存元 → 依存先」の向きか確認する
2. 本文中の `→` を検索し、図と同じ依存方向を示しているか照合する
3. まとめスライドの箇条書きでも方向を確認する（見落としやすい）

## 7. 図だけスライドの検出

Mermaid 図や画像がメインのスライドで、図の要素が十分に説明されているかを確認する。

### 判定基準

図スライドが以下の **両方** を満たす場合は「図だけスライド」として NG:

1. 本文が `▶` 行（So What）しかない（テーブル・箇条書き・コードブロックがない）
2. 後続スライドで図の各要素を個別に解説していない

### 修正方法

| 状況 | 対応 |
|------|------|
| 図の要素を 1 枚で説明できる | テーブルまたは箇条書きを追加（図は削除してよい） |
| 図が大きく補足テキストと共存できない | 概観図 + 解説スライドの 2 枚に分割 |
| 後続で各要素を個別解説している | 概観図として OK — 修正不要 |

Before/After:

| Before (NG) | After (OK) | 理由 |
|---|---|---|
| `## 全体構成` + Mermaid 図 + `▶ A と B で構成される` | `## 全体構成` + 比較テーブル（A vs B の役割・データ・結果）+ `▶ A と B が循環する` | 図だけでは A と B の違いが伝わらない |
| `## データフロー` + Mermaid 図のみ | `## データフロー` + Mermaid 図 + 箇条書き 3 点で各段階を説明 | 矢印の意味が本文にない |

## 8. 最終チェック

以下を満たしたら明瞭性チェック完了:

- 連続 2 スライド以上で同じ主張を繰り返していない
- テキストのみの説明スライドが連続していない
- 見出しと本文の主張が一致している
- 矢印表記（`→` / `-->`）の方向がスライド全体で一貫している
